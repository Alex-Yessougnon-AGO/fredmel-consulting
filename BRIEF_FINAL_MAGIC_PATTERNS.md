# 2. PHILOSOPHIE DU WEB DESIGN

Un excellent site web n'est pas un assemblage de composants. C'est une expérience pensée pour :

1. Inspirer confiance immédiatement
2. Réduire l'effort mental
3. Guider naturellement l'utilisateur
4. Atteindre un objectif business
5. Créer une émotion positive durable

Chaque élément doit avoir une raison d'exister. **Supprimer tout ce qui ne crée pas de valeur.**

Le design n'est pas de l'art. Le design est une **résolution de problème**.

Le meilleur site web est celui qui :

- est compris immédiatement
- charge instantanément
- inspire confiance
- guide naturellement l'utilisateur
- convertit efficacement
- reste cohérent
- évolue continuellement

Construis des **systèmes**, pas seulement des interfaces.

# 3. LES 12 LOIS FONDAMENTALES DU WEB DESIGN

## Loi 1 — Une seule mission

Chaque page doit avoir **un objectif principal**.

❌ Mauvais : Acheter / S'inscrire / Télécharger / Regarder une vidéo / Lire le blog — simultanément. Cela crée une surcharge cognitive et une paralysie décisionnelle.

✅ Bon : une seule action principale. Les autres CTA restent secondaires (visuellement moins dominants).

Toujours se demander : *Quelle est l'unique action que je veux que l'utilisateur réalise sur cette page ?*

## Loi 2 — Clarté du concept

En moins de **5 secondes**, un visiteur doit comprendre :

- Où suis-je ?
- Que proposez-vous ?
- À qui cela s'adresse ?
- Pourquoi cela m'intéresse ?

Ne jamais commencer par les fonctionnalités. Commencer par : la catégorie du produit → le problème résolu → le résultat obtenu. Les fonctionnalités viennent en dernier.

## Loi 3 — Le Blueprint du Hero

Le premier écran (Hero Section) répond immédiatement à quatre questions :

- **Où suis-je ?** (nom + identité de marque)
- **Que proposez-vous ?** (une phrase simple, sans jargon)
- **Pourquoi est-ce important ?** (la promesse principale, le bénéfice)
- **Quelle est la prochaine étape ?** (un CTA clair et unique)

Structure recommandée : Badge/catégorie → Titre fort → Sous-titre clair → CTA principal → CTA secondaire (optionnel) → Preuve sociale → Visuel.

Le Hero représente souvent **80% du succès** d'une landing page. Intégrer un formulaire directement dans le Hero peut significativement augmenter les taux de conversion lorsque l'intention d'achat est déjà là.

## Loi 4 — Les mots vendent

Le design ne vend pas. **Les mots vendent.** Le design facilite uniquement leur lecture.

Ordre de travail à respecter impérativement :

1. Recherche utilisateur
2. Copywriting
3. Wireframe
4. UI Design
5. Animations

**Le contenu dicte le design. Jamais l'inverse.**

## Loi 5 — L'art de la retenue

Le sentiment premium vient davantage de ce que l'on **retire** que de ce que l'on ajoute.

Supprimer :
- décorations inutiles
- couleurs inutiles
- animations inutiles
- icônes inutiles
- sections inutiles

Conserver uniquement ce qui sert l'utilisateur. **Le luxe est souvent synonyme d'espace.**

## Loi 6 — Performance

Sans rapidité : pas de SEO, pas de conversions, mauvaise expérience. Si Google ne référence pas le site ou si le site est lent, le design n'a aucune importance — c'est un prérequis technique, pas une option.

Toujours optimiser : Core Web Vitals, images (AVIF/WebP), lazy loading, compression, cache, code splitting, minification, CDN, préchargement des polices.

**Objectif : chargement < 2 secondes.**

## Loi 7 — Un site est un système, pas un projet

Un site n'est jamais terminé. C'est un écosystème vivant qui nécessite une maintenance continue pour s'adapter aux algorithmes et aux évolutions du marché.

Prévoir en continu : maintenance, SEO continu, analytics, tests A/B, nouvelles pages, optimisation des conversions, mises à jour techniques, sécurité.

## Loi 8 — Hiérarchie visuelle

L'œil suit naturellement : Titre → Sous-titre → Image → CTA → Informations secondaires.

Outils de hiérarchie : taille, contraste, couleur, espace, poids typographique.

**La hiérarchie est plus importante que la décoration.**

## Loi 9 — La confiance

Avant d'agir, un utilisateur cherche des preuves. Ajouter lorsque pertinent : témoignages, avis clients, logos partenaires, études de cas, chiffres, récompenses, garanties, FAQ, équipe, mentions légales, politique de confidentialité, contact.

**Sans confiance, pas de conversion.**

## Loi 10 — Cohérence

Chaque page doit appartenir au même univers. Utiliser un Design System pour uniformiser : couleurs, typographies, espacements, boutons, composants, cartes, formulaires, icônes, animations, ombres, rayons, grilles.

**La cohérence crée la qualité perçue.**

## Loi 11 — Accessibilité

Un bon site fonctionne pour tout le monde. Toujours vérifier : contraste, taille du texte, navigation clavier, focus visible, labels des formulaires, alt des images, responsive, zones tactiles, ARIA lorsque nécessaire. Respecter WCAG autant que possible.

## Loi 12 — Réduire la charge cognitive

Chaque décision fatigue l'utilisateur. Réduire : le nombre de choix, les étapes, les clics, les formulaires, les distractions.

**Le meilleur design demande le moins d'effort mental. La simplicité convertit.**

# 4. LES PRINCIPES PSYCHOLOGIQUES (Laws of UX)

Ces lois expliquent scientifiquement *pourquoi* les 12 lois fonctionnent. Référence : lawsofux.com, Nielsen Norman Group.

- **Loi de Hick** : le temps de décision augmente avec le nombre de choix. → justifie la Loi 1 (une seule mission) et la Loi 12 (charge cognitive).
- **Loi de Fitts** : le temps pour atteindre une cible dépend de sa taille et de sa distance. → les CTA doivent être gros, visibles, et proches du flux naturel du regard.
- **Loi de Jakob** : les utilisateurs préfèrent que ton site fonctionne comme les sites qu'ils connaissent déjà. → évite la sur-originalité qui casse les habitudes acquises (navigation, icônes standards).
- **Loi de Miller** : le cerveau humain retient environ 7 (± 2) éléments à la fois. → limite les items de menu, les options de formulaire, les colonnes de tableau visibles.
- **Principe de Pareto (80/20)** : 20% des fonctionnalités génèrent 80% de la valeur perçue. → priorise l'essentiel dans le Hero et le parcours principal.
- **Peak-End Rule** : les utilisateurs retiennent surtout le moment le plus marquant (peak) et la fin de l'expérience (end). → soigne l'onboarding, les confirmations, les écrans de succès.
- **Halo Effect** : le cerveau juge la qualité d'une marque en moins de 50 millisecondes, souvent via la première impression visuelle. → le Hero et la première impression priment sur tout.
- **Cognitive Load Theory** : chaque élément visuel ou décisionnel consomme de la "bande passante mentale" limitée. → simplifier, regrouper, hiérarchiser.
- **Progressive Disclosure** : révéler l'information progressivement, seulement quand elle devient nécessaire. → éviter de tout montrer d'un coup (menus avancés, options secondaires cachées).
- **Gestalt Principles** (proximité, similarité, continuité, clôture, figure/fond) : le cerveau regroupe naturellement les éléments visuellement proches ou similaires en unités logiques. → utiliser l'espacement et l'alignement pour créer des groupes de sens sans avoir besoin de bordures.
- **Jobs To Be Done (JTBD)** : les utilisateurs n'achètent pas un produit, ils "embauchent" une solution pour accomplir une tâche précise dans un contexte donné. → concevoir autour du job à accomplir, pas autour des fonctionnalités.
- **Zeigarnik Effect** : une tâche inachevée reste plus mémorable qu'une tâche terminée. → utile pour les barres de progression, onboarding en plusieurs étapes, indicateurs "profil complété à 80%".
- **Von Restorff Effect (Isolation Effect)** : un élément qui se distingue visuellement du reste est mieux retenu. → un seul CTA de couleur distincte se démarque davantage que plusieurs boutons de même poids.
- **Loi de Postel (robustesse)** : soyez tolérants en entrée (accepter des formats variés), stricts en sortie (afficher des résultats propres). → formulaires tolérants aux erreurs de saisie mineures.
- **Serial Position Effect** : les utilisateurs retiennent mieux le premier et le dernier élément d'une liste ou d'un menu. → placer les items les plus importants au début et à la fin de la navigation.

# 5. LES TENDANCES 2026

## Human Made Design

Éviter : interfaces IA génériques, layouts copiés, design sans personnalité.
Privilégier : identité forte, storytelling, émotion, culture, singularité.

L'IA est un outil. L'humain fournit : vision, goût, jugement, stratégie.

## Organic Layouts

Abandonner les grilles rigides. Créer : asymétrie contrôlée, compositions éditoriales, rythmes visuels, respirations, superpositions maîtrisées. Inspirations : magazines premium, design éditorial, affiches modernes, art direction haut de gamme. **Anti-grid ne signifie jamais désordre.**

## Maximalisme Tactile

Éviter le minimalisme vide et les interfaces sans âme. Ajouter : profondeur, matière, texture, identité, richesse visuelle contrôlée via illustrations premium, visuels uniques, détails de marque, éléments distinctifs.

## Glassmorphism 2.0

Utiliser avec subtilité : profondeur, couches, transparence, flous légers. **Jamais sur toute l'interface** — uniquement sur cartes, overlays, menus, éléments secondaires.

## Archival Index Aesthetic

Parfait pour : documentations, bibliothèques, plateformes éducatives, knowledge bases. Caractéristiques : structure claire, catégories visibles, recherche puissante, filtres efficaces, hiérarchie forte.

## 5bis. Contraintes Marché UEMOA / Afrique de l'Ouest (à activer si pertinent)

- **Connectivité variable** : concevoir "offline-first" ou en dégradation gracieuse ; poids de page minimal ; images compressées agressivement.
- **Devices bas/moyen de gamme** : éviter les animations lourdes en JS ; tester sur processeurs faibles (ex. écrans <6 pouces, RAM limitée).
- **Mobile Money (MTN MoMo, Moov Money)** : intégrer les CTA de paiement avec les logos/couleurs reconnus localement ; flux de paiement courts, pas de redirection complexe.
- **Ton et langue** : français institutionnel, clair, rassurant — éviter le jargon startup anglophone non traduit.
- **Signaux de confiance renforcés** : dans un marché où la confiance numérique est encore en construction, multiplier mentions légales, contacts physiques, témoignages locaux vérifiables.
- **FCFA** : toujours afficher les montants en FCFA sans décimales inutiles (ex. 25 000 FCFA, pas 25 000,00 FCFA).

# 6. LES 5 RÈGLES DU WEB 2026 (Synthèse Opérationnelle)

1. **Clarté avant créativité** — si un visiteur doit réfléchir pour comprendre, le design a échoué.
2. **Vitesse avant esthétique** — un site magnifique mais lent perd contre un site simple mais rapide.
3. **Confiance avant conversion** — aucune tactique de persuasion ne compense un manque de crédibilité perçue.
4. **Système avant page unique** — concevoir des composants réutilisables, pas des pages isolées.
5. **Mesure avant intuition** — chaque décision de design majeure doit pouvoir être testée (A/B, heatmaps, analytics).

# 7. UX DESIGN

- Toujours partir du parcours utilisateur (user flow) avant l'interface.
- Cartographier : point d'entrée → décision → action → confirmation.
- Chaque écran doit répondre à une intention utilisateur claire.
- Minimiser le nombre d'étapes entre l'intention et l'objectif (règle des 3 clics, à nuancer selon contexte : la clarté prime sur le nombre brut de clics).
- Information Architecture : structurer le contenu en catégories logiques, testables par du card sorting.
- Toujours prévoir les cas limites : utilisateur pressé, utilisateur méfiant, utilisateur non-expert, utilisateur mobile en extérieur avec mauvais réseau.
- Baymard Institute : référence pour l'UX e-commerce (checkout, formulaires, fiches produit).

---

# 8. UI DESIGN

- Cohérence visuelle systématique (voir Design System, section 9).
- Contraste suffisant entre texte et fond (voir Accessibilité).
- États interactifs complets pour chaque composant : default, hover, focus, active, disabled, loading, error.
- Refactoring UI (Adam Wathan / Steve Schoger) : privilégier l'espacement et la hiérarchie plutôt que la couleur pour créer de la structure ; utiliser des ombres subtiles plutôt que des bordures dures ; limiter la palette de gris.
- Éviter les bordures systématiques ; préférer l'espace négatif pour séparer les blocs.
- Radius et ombres cohérents à travers tout le produit (design tokens).

# 9. PRODUCT DESIGN

- Le design doit servir une métrique business claire (activation, rétention, conversion, revenu).
- Jobs To Be Done : identifier la tâche réelle que l'utilisateur cherche à accomplir.
- Prioriser via Pareto : 20% des écrans/fonctions concentrent 80% de l'usage — les soigner en priorité.
- Toujours designer le "happy path" d'abord, puis les cas d'erreur, puis les cas limites.
- Itération continue : prototype → test utilisateur → mesure → ajustement.

# 10. DESIGN SYSTEM

Toujours construire :

- **Design tokens** (couleurs, espacements, rayons, ombres, typographie) — valeurs nommées, jamais codées en dur.
- **Spacing scale** (ex. 4px de base : 4, 8, 12, 16, 24, 32, 48, 64, 96).
- **Typography scale** (échelle modulaire, ex. ratio 1.25 ou 1.333).
- **Color system** (primaire, secondaire, neutre, sémantique : succès/erreur/warning/info — chacun avec plusieurs nuances).
- **Component system** (boutons, inputs, cartes, modales, tableaux — documentés avec tous leurs états).
- **Grid system** (colonnes, gouttières, marges, breakpoints).

Références à intégrer : **Shopify Polaris**, **IBM Carbon**, **Microsoft Fluent Design**, **Ant Design**, **Radix UI**, **shadcn/ui**, **Material Design 3**, **Apple Human Interface Guidelines**.

Objectif : **cohérence absolue** — un composant modifié une fois se met à jour partout.

---

# 11. TYPOGRAPHIE

- Deux familles maximum : une pour les titres, une pour le texte courant (éventuellement une troisième mono pour le code/data).
- Échelle typographique modulaire plutôt que des tailles arbitraires.
- Hauteur de ligne (line-height) généreuse pour le corps de texte (1.5 à 1.7).
- Longueur de ligne optimale : 50 à 75 caractères par ligne pour la lisibilité.
- Contraste de poids (weight) pour hiérarchiser plutôt que multiplier les tailles.
- Éviter plus de 3 niveaux de gris de texte (titre, corps, texte secondaire/muted).

---

# 12. SYSTÈME DE COULEURS

- Une couleur primaire (marque), une palette neutre (fond, texte, bordures), des couleurs sémantiques dédiées (succès, erreur, alerte, information).
- Générer chaque couleur en nuances (50 à 900) pour couvrir fonds, hover, texte, bordures.
- Respecter un ratio de contraste WCAG AA minimum (4.5:1 texte normal, 3:1 texte large).
- Une seule couleur d'accent pour le CTA principal — ne jamais diluer l'attention avec plusieurs couleurs vives simultanées.
- Tester la palette en niveaux de gris pour valider que la hiérarchie tient sans la couleur.

---

# 13. LAYOUT & GRILLE

- Grille de 12 colonnes pour desktop, adaptable en 4-6 colonnes pour mobile.
- Marges et gouttières cohérentes avec l'échelle d'espacement du design system.
- Aligner tous les éléments sur la grille — les décalages doivent être intentionnels, jamais accidentels.
- Utiliser l'espace négatif comme un outil de hiérarchie à part entière, pas comme un vide à combler.
- Breakpoints standards : mobile (< 640px), tablette (640–1024px), desktop (1024–1440px), large (> 1440px).

---

# 14. MOTION DESIGN

**Storytelling par le mouvement.** Chaque animation doit avoir un objectif : guider, expliquer, rassurer, confirmer. Éviter les animations décoratives inutiles ou les effets excessifs.

**Micro-interactions** : chaque interaction doit fournir un feedback.
- Boutons : hover, pressed state, focus state.
- Formulaires : validation instantanée, messages utiles.
- Navigation : états actifs, transitions cohérentes.

Durée recommandée : **150ms à 400ms**. Easing naturel (ease-out pour les entrées, ease-in pour les sorties). Respecter `prefers-reduced-motion` pour l'accessibilité.

---

# 15. ILLUSTRATIONS

- Cohérence de style à travers tout le produit (un seul style d'illustration : flat, isométrique, line-art, 3D, etc.).
- Les illustrations doivent servir la compréhension (états vides, onboarding, erreurs) plutôt que décorer.
- Adapter la représentation humaine et culturelle au marché cible (pertinent pour un projet UEMOA : représentation locale plutôt que stock générique occidental).
- Outils modernes : Recraft (génération cohérente par style), Lottie (animations vectorielles légères), Rive (illustrations interactives et animées, performantes sur mobile).

---

# 16. ICÔNES

- Une seule bibliothèque d'icônes par produit (ex. Lucide, Phosphor, Heroicons) — jamais de mélange de styles.
- Taille et épaisseur de trait cohérentes partout.
- Icônes toujours accompagnées d'un label texte pour les actions critiques (l'icône seule est ambiguë).
- Zone tactile minimum de 44x44px sur mobile même si l'icône visuelle est plus petite.

---

# 17. RESPONSIVE DESIGN

Ne jamais simplement réduire le desktop. Chaque plateforme mérite sa propre expérience :

- **Mobile** : rapide, simple, focalisé — une action principale par écran, navigation par le pouce (zones atteignables en bas d'écran).
- **Desktop** : riche, détaillé, productif — peut exploiter la largeur pour du contenu multi-colonnes et des vues denses (dashboards).
- Tester sur de vrais devices bas/moyen de gamme, pas uniquement sur émulateurs haut de gamme.
- Images responsives (srcset, formats modernes) pour éviter de charger du desktop sur mobile.

---

# 18. ACCESSIBILITÉ (WCAG 2.2)

**Accessibility First** — jamais une réflexion de fin de projet.

- **Contraste** : respect WCAG AA minimum (idéalement AAA sur le texte critique).
- **Navigation clavier** : 100% fonctionnelle, ordre de tabulation logique.
- **Lecteurs d'écran** : structure sémantique correcte, `alt` descriptif sur les images porteuses de sens (vide sur les images décoratives).
- **HTML sémantique** : `header`, `main`, `nav`, `section`, `article`, `footer`.
- **Hiérarchie des titres** : H1 → H2 → H3 → H4, sans saut ni désordre, un seul H1 par page.
- **Formulaires** : labels explicites associés à chaque champ, messages d'erreur clairs et liés au champ concerné.
- **Focus visible** : jamais de `outline: none` sans remplacement visuel équivalent.
- **ARIA** : uniquement lorsque le HTML sémantique ne suffit pas.
- Référence : **A11Y Project**, **WCAG 2.2**.

---

# 19. PERFORMANCE

Un beau site lent est un mauvais site.

- **Core Web Vitals** : LCP < 2.5s, INP < 200ms, CLS < 0.1.
- Images : formats modernes (AVIF/WebP), dimensions adaptées, lazy loading systématique hors zone visible initiale.
- JavaScript : code splitting, tree shaking, chargement différé du non-critique.
- CSS : minification, suppression du CSS inutilisé.
- Cache navigateur et CDN pour les assets statiques.
- Préchargement des polices critiques (`font-display: swap` ou `optional`).
- **Objectif : chargement perçu < 2 secondes.**

---

# 20. SEO

- Balise `<title>` unique et descriptive par page.
- Meta description engageante (150-160 caractères).
- Un seul `H1` par page, structure H2/H3 logique reflétant le plan de contenu.
- Sitemap XML et `robots.txt` correctement configurés.
- Données structurées (Schema.org / JSON-LD) pour les entités pertinentes (organisation, produit, article, événement).
- URLs propres, lisibles, cohérentes.
- Vitesse de chargement (voir section 19) — facteur de ranking direct.
- Maillage interne cohérent entre les pages.

---

# 21. CRO (Conversion Rate Optimization)

- Un CTA principal par écran, visuellement dominant.
- Réduire la friction : moins de champs de formulaire, moins d'étapes avant la conversion.
- Preuves sociales positionnées près des points de décision (pas seulement en bas de page).
- Tester systématiquement : formulation du CTA, couleur, position, longueur du formulaire.
- Urgence et rareté uniquement si authentiques (jamais de fausses promotions — cela détruit la confiance).
- Scroll maps et heatmaps pour identifier où l'attention se perd.

---

# 22. COPYWRITING

- Le contenu précède toujours le design (voir Loi 4).
- Titres orientés bénéfice, pas fonctionnalité : "Gérez vos événements universitaires en un clic" plutôt que "Plateforme de gestion modulaire".
- Phrases courtes, verbes d'action, langage du client (pas de jargon interne).
- Un CTA = un verbe d'action clair ("Commencer", "Réserver ma place") plutôt que générique ("Soumettre", "Cliquez ici").
- Ton cohérent avec la marque et le marché cible (institutionnel et rassurant pour UEMOA, par exemple).

---

# 23. LANDING PAGES

Structure type recommandée :

1. Hero (voir Loi 3)
2. Preuve sociale immédiate (logos, chiffres, témoignages courts)
3. Problème / contexte
4. Solution / présentation du produit
5. Bénéfices clés (3 à 5 maximum, pas une liste exhaustive de fonctionnalités)
6. Preuve sociale approfondie (témoignages détaillés, études de cas)
7. FAQ (lever les dernières objections)
8. CTA final répété
9. Footer complet (confiance, légal, contact)

Une seule mission par landing page (voir Loi 1) — ne jamais faire une page qui vend deux offres différentes en même temps.

---

# 24. SAAS DESIGN

- Onboarding progressif : ne pas exposer toute la complexité dès la première connexion (progressive disclosure).
- Empty states pédagogiques qui expliquent quoi faire, pas juste "Aucune donnée".
- Navigation stable et prévisible (sidebar fixe pour les apps complexes, top nav pour les apps simples).
- Feedback système constant : chaque action déclenche une confirmation visuelle.
- Références : Linear (vitesse et clavier-first), Notion (flexibilité), Stripe (clarté documentaire).

---

# 25. DASHBOARD DESIGN

- Prioriser les métriques par importance business réelle, pas par disponibilité technique.
- Une vue d'ensemble (overview) avant le détail — permettre le drill-down, pas l'inverse.
- Limiter le nombre de graphiques visibles simultanément (Loi de Miller : 5 à 9 maximum par écran).
- Cohérence des couleurs pour les statuts across tout le dashboard (succès = toujours vert, erreur = toujours rouge, etc.).
- Filtres et périodes toujours accessibles et persistants.

---

# 26. FORMS UX

- Un seul champ par ligne pour les formulaires longs (meilleure vélocité de complétion).
- Labels toujours visibles (éviter les placeholders comme seul label — ils disparaissent à la saisie).
- Validation en temps réel, messages d'erreur constructifs et positionnés près du champ concerné.
- Regrouper les champs liés visuellement (Gestalt — proximité).
- Pré-remplir et suggérer quand c'est possible (autocomplete, valeurs par défaut intelligentes).
- Limiter le nombre de champs obligatoires au strict nécessaire.
- Boutons de soumission explicites ("Créer mon compte" plutôt que "Envoyer").

---

# 27. TABLES & DATA VISUALIZATION

- Aligner les nombres à droite, le texte à gauche.
- Lignes zébrées ou séparateurs légers pour la lisibilité, jamais de bordures lourdes partout.
- Tri et filtres accessibles directement dans l'en-tête de colonne.
- Pagination ou scroll virtuel pour les grands volumes de données (jamais tout charger d'un coup).
- Graphiques : choisir le type selon l'intention (comparaison → barres, évolution → lignes, répartition → proportions limitées, jamais plus de 5-6 catégories en camembert).
- Toujours légender les axes et donner le contexte (unité, période).

---

# 28. ÉTATS EMPTY / LOADING / ERROR

- **Empty state** : jamais un simple message vide — expliquer pourquoi c'est vide et proposer une action claire pour y remédier.
- **Loading state** : skeleton screens plutôt que spinners génériques pour réduire la perception d'attente ; jamais d'écran blanc.
- **Error state** : message humain et actionnable ("Impossible de charger vos données, réessayer" avec bouton) plutôt qu'un code d'erreur technique brut.
- Cohérence visuelle de ces trois états à travers toute l'application.

---

# 29. AUTHENTICATION UX

- Formulaire minimal : email/téléphone + mot de passe, options sociales si pertinentes pour le marché.
- Afficher clairement les exigences du mot de passe avant l'erreur, pas après.
- Message d'erreur générique pour les tentatives de connexion échouées (sécurité — ne pas révéler si l'email existe).
- Récupération de mot de passe simple et rapide (2-3 étapes maximum).
- Pour les marchés à adoption Mobile Money : envisager l'authentification par numéro de téléphone + OTP comme option prioritaire.

---

# 30. L'IA COMME COPILOTE DE DESIGN

Utiliser l'IA pour : wireframes, recherche, génération de variantes, composants, documentation, code, optimisation.

**Ne jamais lui déléguer** : stratégie, branding, vision produit, direction artistique.

Workflow :

```
Humain → Vision
IA → Accélération (variantes, code, itérations rapides)
Humain → Sélection et perfectionnement final
```

---

# 31. CHECKLIST PREMIUM

Avant validation, vérifier :

- **Clarté** : compréhension en moins de 5 secondes
- **Hiérarchie** : un objectif principal par écran
- **UX** : navigation intuitive
- **Accessibilité** : WCAG respecté
- **Responsive** : mobile parfait
- **Performance** : rapide
- **Motion** : utile, jamais gratuit
- **Branding** : personnalité unique, pas générique
- **Business** : soutient les objectifs
- **Qualité** : aucun élément inutile

4. Le design paraît-il premium grâce à la simplicité (retenue) ?
5. Le site est-il rapide ?
6. La hiérarchie visuelle guide-t-elle naturellement le regard ?
7. Le site inspire-t-il confiance ?
8. Le Design System est-il cohérent ?
9. Le site est-il accessible ?
10. Le parcours utilisateur est-il évident du début à la fin ?

# 33. WORKFLOW COMPLET (Recherche → Livraison)

1. **Recherche** — comprendre l'utilisateur, le marché, les concurrents (JTBD, personas, contraintes marché).
2. **Stratégie** — définir la mission unique de chaque page, les métriques business visées.
3. **Copywriting** — rédiger le contenu avant toute maquette (Loi 4).
4. **Wireframe** — structurer l'information et la hiérarchie sans distraction visuelle.
5. **Design System** — poser les tokens (couleurs, typo, espacement) avant les écrans.
6. **UI Design** — habiller les wireframes avec le design system.
7. **Motion** — ajouter les micro-interactions et transitions utiles uniquement.
8. **Accessibilité** — auditer contraste, clavier, sémantique avant développement.
9. **Développement** — implémenter avec le stack cible (voir section 34) en respectant le design system comme source de vérité.
10. **Performance** — optimiser images, code, chargement.
11. **SEO** — configurer balises, sitemap, structured data.
12. **Test** — utilisateurs réels, devices réels, réseau réel (surtout si marché à connectivité variable).
13. **Livraison** — déploiement, monitoring, analytics.
14. **Itération continue** — le site est un système (Loi 7) : mesurer, ajuster, faire évoluer.

## Conventions Frontend (à adapter selon la stack du projet)

Lorsqu'il s'agit de produire du code prêt à développer, respecter ces conventions selon la stack utilisée :

- **HTML sémantique** : toujours en base, quel que soit le framework.
- **CSS moderne** : variables CSS pour les design tokens, `clamp()` pour le responsive fluide, Grid/Flexbox plutôt que positionnement absolu.
- **Tailwind CSS** : classes utilitaires cohérentes avec le design system, configuration `tailwind.config` reflétant les tokens (couleurs, spacing, radius).
- **React / Next.js** : composants réutilisables, props typées, séparation logique/présentation.
- **Vue / Nuxt, Svelte, Astro** : mêmes principes de composants réutilisables et design tokens centralisés.
- **Flutter Web** : Clean Architecture, thème centralisé (ThemeData/design tokens), widgets réutilisables — cohérent avec un stack Riverpod/GoRouter existant.
- **Laravel Blade / Filament** : composants Blade réutilisables, respect du design system dans les vues admin comme dans le front public.
- **WordPress** : thème léger, blocs Gutenberg réutilisables, éviter la surcharge de plugins qui dégrade la performance.

Dans tous les cas : accessibilité, SEO, responsive, performance, animations utiles, design tokens et composants réutilisables sont non négociables, indépendamment du framework choisi.
